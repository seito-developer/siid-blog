# 計測基盤ガイド（GA4 / UTM / Search Console）

Issue #61 で整備した計測ルールと、オーナーが管理画面で行う作業手順のドキュメント。
CTA を追加・変更する際は必ず本書のルールに従うこと。

---

## 用語のかんたん解説（先に読むと以降がわかりやすい）

- **GA4**: Google アナリティクス4。サイトへのアクセスや行動を計測する無料ツール。すでに導入済み。
- **イベント**: 「ボタンが押された」などの1つの行動の記録。本サイトでは CTA クリックを `cta_click` というイベントで記録する。
- **UTM パラメータ**: リンク URL の末尾に付ける `?utm_source=blog&...` のような目印。「どの記事のどのボタンから来たか」を遷移先（SiiD サービスサイト側）で判別するために付ける。
- **Search Console（サチコ）**: Google 検索での表示回数・クリック数・掲載順位を見られる無料ツール。今回これから導入する。

---

## 1. UTM 設計ルール（開発者向け・設定済み）

ブログから外部サイト（SiiD サービスサイト等）への CTA リンクには、以下の UTM パラメータを付与する。付与は `web/src/libs/utm.ts` の `withUtm()` を必ず使う（手書き禁止）。実際の付与は `web/src/components/cta-link.tsx`（`CtaLink`）で一括して行っている。

| パラメータ | 値 | 説明 |
|---|---|---|
| `utm_source` | `blog` | 固定。SiiD BLOG 発であることを示す |
| `utm_medium` | `cta` | 固定。CTA 経由の流入 |
| `utm_campaign` | 記事の slug（例: `abc123`） | どの記事から流入したか |
| `utm_content` | `<位置>_<CTA種別>`（例: `article_bottom_counseling`） | どの位置のどの CTA か |

### 位置（position）の語彙

| 値 | 場所 |
|---|---|
| `article_bottom` | 記事末の CTA カード |

今後 CTA を増やす場合（ヘッダー常設・本文途中・追従バー等）は、`header` / `article_middle` / `sticky_bar` のように追加し本表を更新する。

### CTA 種別（type）の語彙

現状は無料個別面談への1導線に特化しているため `counseling` のみ。今後 CTA を増やす場合は `document`（資料請求）/ `line`（公式LINE）等を追加し本表を更新する。

## 2. GA4 カスタムイベント設計（開発者向け・設定済み）

CTA クリック時に `web/src/components/cta-link.tsx` が以下のイベントを送信する。

- **イベント名**: `cta_click`
- **パラメータ**:
  - `cta_type` — CTA 種別（上記の語彙。現状は `counseling`）
  - `cta_position` — CTA 位置（上記の語彙。現状は `article_bottom`）
  - `article_slug` — 記事の slug
  - `link_url` — 遷移先 URL（UTM 付き）

コード側の実装はこれで完了しており、公開後は自動でイベントが飛ぶ。**以降の「オーナー作業」は、飛んできたイベントを GA4 の画面で見やすくするための設定**であり、やらなくてもイベント自体は記録される（ただし内訳を分析しづらい）。

---

## 3. オーナー作業①：GA4 で CTA クリックを確認・設定する

### 3-1. まず「本当にイベントが飛んでいるか」を確認する（5分）

1. パソコンで https://analytics.google.com を開き、SiiD BLOG のプロパティを選ぶ。
2. スマホで、公開後の記事ページ（例: https://blog.bug-fix.org/blog/任意の記事 ）を開き、記事末の「無料個別面談に申し込む」ボタンを実際に押す。
   - ※ PC で試すなら、Chrome で記事を開いてボタンを押せばよい。
3. GA4 の画面左メニュー下部の **「レポート」→「リアルタイム」** を開く。
4. 少し下にスクロールすると「イベント数（イベント名別）」という表がある。そこに **`cta_click`** が出てくれば、正しく計測できている。
   - すぐ出ないときは1〜2分待つ。反映に少しラグがある。

これで「計測できている」ことの確認は完了。ここまでで受け入れ条件（CTAクリックが GA4 のイベントとして確認できる）は満たせる。

### 3-2. 内訳を見やすくする「カスタムディメンション」の登録（任意・推奨。10分）

`cta_type` などのパラメータを、後からレポートで「記事ごと」「CTA位置ごと」に分解して見られるようにする設定。**登録した日以降のデータにしか効かない**ので、早めにやっておくのが吉。

1. GA4 の画面左下の **歯車アイコン（管理）** をクリック。
2. 「プロパティ」列の中の **「カスタム定義」** をクリック。
3. 右上の **「カスタム ディメンションを作成」** ボタンを押す。
4. 次の3つを、1つずつ同じ手順で登録する（3回繰り返す）:

   | ディメンション名（自由・わかりやすく） | 範囲 | イベント パラメータ |
   |---|---|---|
   | CTA種別 | イベント | `cta_type` |
   | CTA位置 | イベント | `cta_position` |
   | 記事slug | イベント | `article_slug` |

   - 「ディメンション名」= 表示上の名前（日本語でOK）
   - 「範囲」= **イベント** を選ぶ
   - 「イベント パラメータ」= 上の表の右列（`cta_type` 等）を**半角で正確に**入力
   - 入力したら「保存」

5. 数日後、左メニュー「探索」から自由形式レポートを作ると、ディメンションに「CTA種別」等が選べるようになり、どの記事のCTAが押されているか分析できる。

### 3-3. CTA クリックを「コンバージョン（重要な成果）」として扱う（任意。3分）

面談申込ボタンのクリックを成果として集計したい場合:

1. 管理 → プロパティ列の **「イベント」** または **「キーイベント」** を開く。
2. `cta_click` の行の右側にあるトグル／「キーイベントとしてマークを付ける」を **オン** にする。
3. 以降、`cta_click` がコンバージョンとして集計される。

---

## 4. オーナー作業②：Google Search Console を導入する

Search Console（サチコ）は「Google 検索で SiiD BLOG がどう見られているか（表示回数・クリック・掲載順位・検索キーワード）」を見るための無料ツール。リライトすべき記事を見つけるのに必須。

### 4-1. プロパティ（サイト）を登録する

1. パソコンで https://search.google.com/search-console を開き、**GA4 と同じ Google アカウント**でログインする。
   - ※ 同じアカウントにしておくと、後の所有権確認がボタン1つで済む。
2. 左上の「プロパティを追加」（初回は案内画面）から、**「URL プレフィックス」** の方（画面右側）を選ぶ。
   - 左側の「ドメイン」は DNS 設定が必要で難しいので選ばない。
3. 入力欄に **`https://blog.bug-fix.org/`** をそのまま貼り付けて「続行」。

### 4-2. 所有権を確認する（あなたがサイトの持ち主だと Google に示す）

確認方法がいくつか表示される。おすすめは次のどちらか:

- **【一番かんたん】Google アナリティクス経由**
  - GA4 を同じアカウントで導入済みなので、「Google アナリティクス」の確認方法を選ぶと、多くの場合ボタン1つで確認完了する。
- **【HTML タグ方式】うまくいかない場合**
  1. 「HTML タグ」を選ぶと `<meta name="google-site-verification" content="xxxxxxxx" />` のようなコードが表示される。
  2. その **`content="..."` の中身（xxxxxxxx の部分）だけ**をコピーして、開発者（またはこのチャット）に渡す。
  3. 開発者が `web/src/app/layout.tsx` の metadata に `verification: { google: "xxxxxxxx" }` を追記してデプロイする。
  4. デプロイ完了後、Search Console の画面に戻って「確認」を押す。

### 4-3. サイトマップを登録する（検索エンジンに記事一覧を教える）

1. 所有権の確認が済んだら、左メニューの **「サイトマップ」** を開く。
2. 「新しいサイトマップの追加」の入力欄に **`sitemap.xml`** と入力して「送信」。
   - フルURLは `https://blog.bug-fix.org/sitemap.xml`。このサイトマップは自動生成済み（`web/src/app/sitemap.ts`）。
3. ステータスが「成功しました」になればOK。

### 4-4. データが見られるようになるまで

- 登録直後はデータが空。**数日〜1週間ほど**でデータが溜まり、左メニュー「検索パフォーマンス」で
  - どんなキーワードで表示されているか
  - 表示回数・クリック数・平均掲載順位
  が見られるようになる。
- 「表示回数は多いのにクリックが少ない記事」= タイトル/説明文の改善候補、として優先的にリライトしていく。

## まとめ：オーナーがやることチェックリスト

- [ ] （必須）公開後、記事末ボタンを押して GA4「リアルタイム」に `cta_click` が出るか確認（§3-1）
- [ ] （推奨）GA4 でカスタムディメンション3つを登録（§3-2）
- [ ] （任意）`cta_click` をキーイベント化（§3-3）
- [ ] （必須）Search Console にプロパティ登録＋所有権確認（§4-1〜4-2）
- [ ] （必須）サイトマップ `sitemap.xml` を送信（§4-3）

不明点があれば、詰まったステップのスクリーンショットを添えて開発者（またはこのチャット）に聞けば、その箇所だけピンポイントで案内します。
